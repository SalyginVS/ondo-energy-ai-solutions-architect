## Category 1: Core AI/ML Concepts
*(14 терминов)*

**1. Neural Network (Нейронная сеть)**
- **Definition:** Математическая модель, вдохновлённая структурой мозга. Состоит из слоёв узлов (нейронов), передающих взвешенные сигналы.
- **Mechanism:** Обучение = итеративная корректировка весов через backpropagation (обратное распространение ошибки) для минимизации функции потерь.
- **VS Energy Application:** Все LLM в лаборатории — нейронные сети. Понимание структуры объясняет, почему модели "не знают" данных после cutoff даты и требуют RAG для актуальной информации.

**2. Parameters / Weights (Параметры / Веса)**
- **Definition:** Числовые значения внутри нейронной сети, определяющие её поведение. "Знания" модели закодированы в весах.
- **Mechanism:** 30B параметров = 30 миллиардов чисел. В FP16 = 60GB. После Q4-квантизации ≈ 18GB.
- **VS Energy Application:** Размер параметров напрямую определяет требования к VRAM. Основа для понимания таблицы quantization в Category 6.

**3. Training (Обучение)**
- **Definition:** Процесс формирования весов модели на больших массивах данных. Меняет веса модели.
- **Mechanism:** Требует сотни/тысячи GPU часов и терабайты данных. Однократная операция для базовой модели.
- **VS Energy Application:** VS Energy не занимается pre-training. Актуальные подходы: Fine-tuning (Lab A4) или RAG (Lab A3) — оба значительно дешевле.

**4. Inference (Вывод)**
- **Definition:** Процесс генерации ответа уже обученной моделью. Не меняет веса.
- **Mechanism:** Прямой проход через нейронную сеть (forward pass) для каждого токена.
- **VS Energy Application:** Всё что происходит в on-premise лаборатории — это inference. On-premise inference = независимость от внешних API даже при нестабильной сети или инцидентах ИБ.

**5. Transformer**
- **Definition:** Базовая архитектура нейросетей для обработки последовательностей (текст, код, временные ряды), основанная на механизме внимания (attention).
- **Mechanism:** Параллельная обработка токенов с квадратичной сложностью O(n²) от длины контекста.
- **VS Energy Application:** Фундамент всех LLM в лаборатории. Квадратичная сложность = контроль объёма SAP-выгрузок во избежание OOM.

**6. Self-Attention (Само-внимание)**
- **Definition:** Механизм, вычисляющий степень взаимной важности каждой пары токенов в последовательности.
- **Mechanism:** Создаёт матрицу весов размером (n × n). При контексте 100K токенов — матрица 10 миллиардов элементов. Главный потребитель VRAM через KV-Cache.
- **VS Energy Application:** Объясняет, почему модель "понимает" связи в длинном юридическом договоре на 80 страниц, но потребляет при этом значительную часть 24GB VRAM.

**7. Mixture of Experts / MoE (Смесь экспертов)**
- **Definition:** Архитектура, разбивающая нейронную сеть на специализированные блоки (эксперты) и маршрутизатор (router).
- **Mechanism:** Активирует только 2-4 эксперта из десятков для каждого токена. Экономит FLOPs (вычисления), но НЕ экономит VRAM — все веса всех экспертов загружены в память. Работа на RTX 3090 обеспечивается квантизацией, а не MoE.
- **VS Energy Application:** qwen3-coder:30b и qwen3:30b — MoE-модели. Дают 3-4× скорость генерации по сравнению с dense-аналогом. Риск: "коллапс маршрутизатора" на задачах Python + Modbus SCADA одновременно.

**8. Dense Model (Плотная модель)**
- **Definition:** Классическая архитектура, где все параметры активируются при обработке каждого токена.
- **Mechanism:** Нет маршрутизатора. Все нейроны участвуют в каждом forward pass.
- **VS Energy Application:** qwen3:14b, deepseek-r1:14b — dense модели. Предпочтительны для SOC-анализа и GRC-аудита, где потеря деталей из-за неверного срабатывания router недопустима.

**9. State Space Models / SSM (Модели пространства состояний)**
- **Definition:** Альтернативная архитектура с линейной сложностью O(n) вместо квадратичной O(n²).
- **Mechanism:** Сжимает историю токенов в состояние фиксированного размера (hidden state). Mamba, Mamba-2, Gated DeltaNet — реализации SSM.
- **VS Energy Application:** Целевая архитектура для будущего анализа непрерывных потоков SCADA-логов, где важна длина контекста, а не точный поиск единичного факта в документе. Qwen3-Next уже использует Hybrid Transformer+SSM.

**10. Tokenization (Токенизация)**
- **Definition:** Разбивка сырого текста на суб-словные единицы (токены) с присвоением числовых ID.
- **Mechanism:** Качество зависит от размера словаря (vocabulary). Кириллица в моделях с маленьким словарём токенизируется с потерей: "тепломеханический" → 8-10 фрагментов. Штраф к эффективному контексту: ×1.5-2.0 для русского/украинского.
- **VS Energy Application:** Обоснование выбора семейства Qwen (словарь 150K+ токенов, оптимизированный под мультиязычность) для обработки документации предприятия на русском и украинском.

**11. Embedding (Векторное представление)**
- **Definition:** Числовой вектор в многомерном пространстве, кодирующий семантический смысл текста.
- **Mechanism:** Семантически близкие тексты → геометрически близкие векторы. Близость измеряется косинусным сходством (Cosine Similarity).
- **VS Energy Application:** Превращает разрозненные мануалы по турбинам и регламенты ПУЭ в математически сравнимые объекты. Основа RAG-системы (Lab A3).

**12. Hallucination (Галлюцинация)**
- **Definition:** Генерация уверенно звучащей, но фактически ложной информации моделью.
- **Mechanism:** Модель оптимизирована на правдоподобность текста, а не на истинность фактов. При отсутствии данных в обучающем корпусе — интерполирует.
- **VS Energy Application:** Критический риск. Галлюцинация параметров нагрузки трансформатора или настроек защитной автоматики может привести к аварии. Обязательна верификация критических outputs через первоисточники.

**13. RLHF (Reinforcement Learning from Human Feedback)**
- **Definition:** Метод обучения через обратную связь от людей для выравнивания поведения модели с предпочтениями пользователя.
- **Mechanism:** Люди ранжируют ответы → обучается модель вознаграждения → LLM обучается максимизировать вознаграждение (PPO-алгоритм).
- **VS Energy Application:** Все instruction-tuned модели в стеке прошли RLHF. Объясняет их "вежливость" и следование инструкциям по сравнению с base-моделями.

**14. Few-shot / Zero-shot Learning**
- **Definition:** Zero-shot — задача без примеров в промпте. Few-shot — задача с 2-5 примерами в промпте.
- **Mechanism:** Модели достаточно нескольких примеров (shots) для понимания паттерна задачи без переобучения.
- **VS Energy Application:** Few-shot с примерами правильно оформленных заявок на изменение (Change Request) позволяет LLM форматировать новые заявки по стандарту VS Energy без fine-tuning.

## Category 2: LLM & Prompt Engineering
*(12 терминов)*

**15. Prompt (Промпт)**
- **Definition:** Входной текст, направляющий поведение языковой модели.
- **Mechanism:** Единственный канал управления моделью на этапе inference без изменения весов.
- **VS Energy Application:** Качество промпта определяет качество вывода. Плохой промпт к LLM = плохой SQL-запрос к базе данных.

**16. System Prompt (Системный промпт)**
- **Definition:** Скрытая инструкция, задающая роль, ограничения и контекст модели перед диалогом с пользователем.
- **Mechanism:** Размещается перед историей диалога. Невидим пользователю в production, но модель учитывает его при генерации.
- **VS Energy Application:** Содержит роль ("Ты ассистент SOC-инженера VS Energy"), ограничения ("Не раскрывай IP-адреса SCADA") и процедуры. НИКОГДА не хардкодить API-ключи и пароли — нарушение LLM07:2025.

**17. Prompt Engineering (Инженерия промптов)**
- **Definition:** Дисциплина создания эффективных инструкций для управления поведением LLM.
- **Mechanism:** Включает техники: role assignment, few-shot examples, chain-of-thought, output format specification.
- **TCO Оценка:** Нулевые затраты на инфраструктуру. Высокие затраты времени инженеров. Быстрый старт без изменения модели.

**18. Chain-of-Thought / CoT (Цепочка рассуждений)**
- **Definition:** Техника промпта, побуждающая модель "думать вслух" перед финальным ответом.
- **Mechanism:** Добавление "Let's think step by step" или структурированных шагов рассуждения значительно повышает точность на сложных задачах.
- **VS Energy Application:** Применять для анализа инцидентов ИБ (RCA — Root Cause Analysis) и расчёта технико-экономических обоснований. Не применять для простых справочных запросов — избыточно.

**19. Context Window (Контекстное окно)**
- **Definition:** Максимальное количество токенов, которые модель видит за один раз (ввод + вывод суммарно).
- **Mechanism:** Заявленный лимит (например, 128K) ≠ реальный лимит на RTX 3090. Реальный лимит ограничен доступной VRAM под KV-Cache после загрузки весов модели.
- **VS Energy Application:** qwen3:14b (9.3GB веса) оставляет ~14GB под KV-Cache = реально ~40-60K токенов при Q8. Всегда тестировать реальный лимит на своём железе.

**20. Temperature (Температура)**
- **Definition:** Параметр управления случайностью генерации (0.0 = детерминированный, >1.0 = хаотичный).
- **Mechanism:** Масштабирует логиты перед softmax. При T=0 всегда выбирается наиболее вероятный токен.
- **VS Energy Application:** 0.0-0.2 для парсинга конфигураций сетевого оборудования, анализа SAP-данных, генерации отчётов. 0.7-0.9 для архитектурных брейнштормов и генерации идей.

**21. Top-P / Nucleus Sampling**
- **Definition:** Параметр, ограничивающий выборку токенами, суммарная вероятность которых ≥ P.
- **Mechanism:** При Top-P=0.9 модель выбирает из минимального набора токенов, покрывающих 90% вероятностной массы.
- **VS Energy Application:** Используется совместно с Temperature. Для технических задач: Top-P=0.9, Temperature=0.1.

**22. Output Parsing (Парсинг вывода)**
- **Definition:** Извлечение структурированных данных из текстового ответа LLM.
- **Mechanism:** JSON-режим, регулярные выражения, или специализированные библиотеки (Pydantic, Outlines).
- **VS Energy Application:** Критично для интеграции LLM с SAP API — модель возвращает текст, SAP ожидает структурированный JSON/XML.

**23. Role Prompting (Ролевой промпт)**
- **Definition:** Присвоение модели роли эксперта для повышения качества ответов в конкретной домене.
- **Mechanism:** "Ты Senior Network Engineer с 15 годами опыта в промышленных сетях" активирует соответствующие паттерны из обучающего корпуса.
- **VS Energy Application:** "Ты эксперт по протоколам SCADA (Modbus, IEC 61850)" улучшает качество анализа конфигураций ОТ-оборудования.

**24. Tokens per Second / t/s (Токенов в секунду)**
- **Definition:** Скорость генерации текста моделью. Основная метрика производительности inference.
- **Mechanism:** Зависит от размера модели, квантизации, длины KV-Cache и пропускной способности памяти GPU.
- **VS Energy Application:** Пороговое значение: <15 t/s — система некомфортна для инженеров. qwen3:14b на RTX 3090 даёт ~40-60 t/s при Q8.

**25. Time to First Token / TTFT (Время до первого токена)**
- **Definition:** Задержка от отправки запроса до появления первого символа ответа.
- **Mechanism:** Зависит от длины входного промпта (prefill-фаза). Длинный промпт = высокий TTFT.
- **VS Energy Application:** Критично для Agentic-систем и автоматических скриптов реагирования на инциденты ИБ. Целевой TTFT для SOC-ассистента: <3 секунды.

**26. Quantization / Квантизация (перенесена в Cat.6, здесь — ссылка)**
- *Подробное описание в Category 6. Здесь: квантизация — единственный механизм, позволяющий запустить 30B-модели на 24GB VRAM. Не путать с MoE (экономит FLOPs, не VRAM).*

## Category 3: Embeddings & RAG
*(10 терминов)*

**27. RAG (Retrieval-Augmented Generation — Генерация с извлечением)**
- **Definition:** Архитектурный паттерн: перед генерацией ответа LLM получает релевантные фрагменты из внешней базы знаний.
- **Mechanism:** Query → Embedding → Vector Search → Retrieved Chunks → LLM Prompt → Answer.
- **TCO Оценка:** Оптимальный ROI. Добавление новых документов = загрузка векторов без переобучения модели. Основные затраты: embedding-вычисления + поддержка векторной БД.
- **VS Energy Application:** Интеграция 10GB корпоративных документов (SAP-договора, регламенты ПУЭ, SCADA-мануалы) без изменения модели. Целевое решение для Lab A3.

**28. Vector Database (Векторная база данных)**
- **Definition:** Специализированное хранилище для embedding-векторов с поддержкой поиска по семантической близости.
- **Mechanism:** Индексирует миллионы векторов, выполняет ANN (Approximate Nearest Neighbor — приближённый поиск ближайших соседей) за миллисекунды.
- **VS Energy Application:** Варианты для on-premise: Milvus (enterprise-grade, production), ChromaDB (легковесный, для разработки). Milvus предпочтителен для production с учётом требований к надёжности критической инфраструктуры.

**29. Chunking (Разбивка на фрагменты)**
- **Definition:** Разделение длинных документов на части перед векторизацией.
- **Mechanism:** Размер chunk влияет на точность (мелкие chunk → потеря контекста) и стоимость (крупные chunk → дороже embedding). Стратегии: fixed-size, sentence, semantic.
- **VS Energy Application:** Для SCADA-мануалов оптимален semantic chunking по разделам. Для юридических договоров SAP — chunking по статьям.

**30. Semantic Search (Семантический поиск)**
- **Definition:** Поиск по смыслу запроса, а не по точному совпадению ключевых слов.
- **Mechanism:** Векторизация запроса → косинусное сходство с векторами документов → ранжирование по релевантности.
- **VS Energy Application:** Инженер пишет "как отключить линию при падении давления" → находится мануал "Процедура аварийного отключения при недопустимом снижении давления масла в трансформаторе".

**31. Cosine Similarity (Косинусное сходство)**
- **Definition:** Метрика близости двух векторов в многомерном пространстве. Значение от -1 до 1.
- **Mechanism:** Измеряет угол между векторами. 1.0 = идентичные, 0.0 = несвязанные, -1.0 = противоположные.
- **VS Energy Application:** Порог релевантности для RAG: >0.75 — включать в контекст LLM, <0.75 — исключать. Ниже порога → риск галлюцинаций из нерелевантного контекста.

**32. Reranking (Переранжирование)**
- **Definition:** Вторичная сортировка retrieved-фрагментов по более точной модели после первичного поиска.
- **Mechanism:** Retrieval возвращает top-20 → Cross-encoder reranker переоценивает и возвращает top-5 наиболее релевантных.
- **VS Energy Application:** Критично при поиске по разнородной базе (технические + юридические + billing документы). Без reranking — риск попадания нерелевантного контекста в промпт.

**33. Hybrid Search (Гибридный поиск)**
- **Definition:** Комбинация векторного (семантического) и ключевого (BM25) поиска.
- **Mechanism:** Ключевой поиск силён на точных терминах (артикулы оборудования, номера договоров). Векторный — на концептуальных запросах.
- **VS Energy Application:** Обязателен для базы документов VS Energy: "договор №VS-2024-1847" (ключевой) + "условия расторжения при форс-мажоре" (семантический).

**34. qwen3-embedding (Модель векторизации)**
- **Definition:** Специализированная модель Alibaba для генерации embedding-векторов, оптимизированная под мультиязычный корпус.
- **Mechanism:** Широкий словарь (150K+ токенов) обеспечивает эффективную токенизацию кириллицы. Технические термины на русском/украинском представлены как единые токены, а не фрагменты.
- **VS Energy Application:** Выбран вместо nomic-embed-text: более плотный семантический вектор для кириллических технических терминов ("тепломеханический" = 1-2 токена vs 8-10 у конкурентов). Уже установлен в Ollama.

**35. Fine-tuning / LoRA / QLoRA**
- **Definition:** Дообучение базовой модели на специализированных данных для улучшения performance в конкретном домене.
- **Mechanism:** LoRA (Low-Rank Adaptation) — добавляет малые обучаемые матрицы к замороженным весам. QLoRA — LoRA + квантизация обучения для экономии VRAM.
- **TCO Оценка:** Высокая стоимость. GPU-часы на обучение + создание качественных датасетов руками экспертов. Не для добавления новых фактов (для этого RAG). Для изменения стиля/формата вывода.
- **VS Energy Application:** Потенциальное применение: обучение модели специфическому синтаксису legacy-оборудования или закрытому формату конфигурации коммутаторов.

**36. Agentic AI / AI Agents (Агентный ИИ)**
- **Definition:** Архитектура, где LLM автономно планирует и выполняет многошаговые задачи с использованием внешних инструментов.
- **Mechanism:** LLM получает задачу → планирует шаги → вызывает tools (API, поиск, код) → анализирует результаты → итерирует до достижения цели.
- **TCO Оценка:** Наивысший уровень сложности и риска. Требует: оркестрации, изоляции API-доступов, human-in-the-loop для критических действий.
- **VS Energy Application:** Цепочка: алерт Zabbix → LLM анализирует → запрашивает логи сервера → формирует тикет Jira с RCA. Требует строгих ограничений прав (LLM06:2025 — Excessive Agency).

## Category 4: Security Taxonomy (OWASP LLM Top 10:2025)
*(10 терминов)*

> ⚠️ **Актуальная версия:** OWASP LLM Top 10:2025 (опубликован ноябрь 2024). Нумерация изменена по сравнению с версией 2023. Использовать только эту нумерацию в документации VS Energy.

**37. LLM01:2025 — Prompt Injection (Инъекция промпта)**
- **Definition:** Атака, при которой злоумышленник вставляет инструкции в обрабатываемый контент для переопределения поведения модели.
- **VS Energy Threat Scenario:** Злоумышленник вставляет скрытый текст (белый шрифт на белом фоне) в PDF-счёт от подрядчика. При обработке счёта через RAG-систему скрытая инструкция заставляет LLM изменить реквизиты получателя оплаты в финальном резюме для бухгалтера.
- **Mitigation:** Валидация и санитизация входных данных, разделение каналов инструкций и данных, sandbox для обработки внешних документов.

**38. LLM02:2025 — Sensitive Information Disclosure (Раскрытие чувствительной информации)**
- **Definition:** Непреднамеренное раскрытие конфиденциальных данных через вывод модели.
- **VS Energy Threat Scenario:** Внутренняя LLM в ответ на невинный вопрос сотрудника выдаёт IP-адреса контроллеров SCADA, пароли сервисных учётных записей или персональные данные клиентов из векторной базы.
- **Mitigation:** Классификация данных перед загрузкой в RAG, output filtering, Role-Based Access Control на уровне векторной БД.

**39. LLM03:2025 — Supply Chain Vulnerabilities (Уязвимости цепочки поставок)**
- **Definition:** Риски от использования скомпрометированных моделей, библиотек или данных из недоверенных источников.
- **VS Energy Threat Scenario:** Загрузка GGUF-файла модели с неверифицированного зеркала Hugging Face со скрытым backdoor, который активируется при обработке специфических паттернов в SCADA-конфигурациях.
- **Mitigation:** Использовать только официальные источники (ollama.com, официальные HF-репозитории авторов). Верификация SHA-256 хешей.

**40. LLM04:2025 — Data and Model Poisoning (Отравление данных и модели)**
- **Definition:** Намеренное искажение обучающих данных или векторной базы знаний для изменения поведения модели.
- **VS Energy Threat Scenario:** Инсайдер вносит изменения в регламенты безопасности в корпоративной базе знаний, из-за чего RAG начинает выдавать технически некорректные процедуры обслуживания оборудования.
- **Mitigation:** Контроль версий (Git) для всех документов в RAG, аудит изменений, разделение прав записи в векторную БД.

**41. LLM05:2025 — Improper Output Handling (Некорректная обработка вывода)**
- **Definition:** Слепое использование сгенерированного кода или команд без верификации.
- **VS Energy Threat Scenario:** LLM генерирует Python-скрипт для внутреннего дашборда мониторинга с уязвимостью SQL-инъекции. DevOps-инженер выполняет код без code review, открывая вектор атаки на SAP-базу данных.
- **Mitigation:** Обязательный human review сгенерированного кода, статический анализ, sandbox для исполнения.

**42. LLM06:2025 — Excessive Agency (Избыточная автономия)**
- **Definition:** Предоставление LLM-агенту прав на действия, превышающих необходимый минимум (нарушение Principle of Least Privilege).
- **VS Energy Threat Scenario:** LLM-агент для помощи администраторам SAP получает права на запись в финансовые модули. Из-за галлюцинации или некорректно интерпретированного запроса агент удаляет критичные проводки за отчётный месяц.
- **Mitigation:** Минимальные права (read-only где возможно), human-in-the-loop для деструктивных операций, аудит всех действий агента, явный whitelist разрешённых операций.

**43. LLM07:2025 — System Prompt Leakage (Утечка системного промпта)** *(НОВОЕ в 2025)*
- **Definition:** Раскрытие конфиденциального системного промпта пользователю через jailbreak или галлюцинацию.
- **VS Energy Threat Scenario:** Сотрудник применяет технику jailbreak и заставляет LLM-ассистента раскрыть системный промпт, в котором захардкожены токены авторизации к внутреннему GitLab или описана топология защищённой сети ОТ-сегмента.
- **Mitigation:** Никогда не хардкодить секреты в системный промпт. API-ключи — только через Vault + Tool Calling. Промпт проектировать с учётом его возможной компрометации.

**44. LLM08:2025 — Vector & Embedding Weaknesses (Уязвимости векторов)** *(НОВОЕ в 2025)*
- **Definition:** Атаки на целостность и конфиденциальность векторной базы знаний RAG-системы.
- **VS Energy Threat Scenario:** Атакующий получает несанкционированный доступ к Milvus и модифицирует embedding-векторы регламентов безопасности подстанций. RAG-система начинает выдавать процедуры, ведущие к аварийным ситуациям.
- **Mitigation:** Сетевая изоляция векторной БД (не экспонировать в общую сеть), шифрование at-rest и in-transit, мониторинг целостности векторов, регулярные бэкапы.

**45. LLM09:2025 — Misinformation (Дезинформация / Галлюцинации)** *(НОВОЕ в 2025)*
- **Definition:** Генерация уверенно звучащей, но фактически ложной информации с риском принятия решений на её основе (overreliance).
- **VS Energy Threat Scenario:** Инженер смены запрашивает у LLM допустимые параметры нагрузки на трансформатор. Модель галлюцинирует завышенные показатели, опираясь на базовые веса, а не на технический паспорт конкретного устройства. Инженер доверяет ответу без верификации и не отключает перегруженную линию, что приводит к аппаратной аварии на подстанции.
- **Mitigation:** RAG с официальными техническими паспортами, обязательная верификация критических outputs через первоисточники, UI-предупреждения о необходимости проверки.

**46. LLM10:2025 — Unbounded Consumption (Неограниченное потребление ресурсов)** *(обновлено в 2025)*
- **Definition:** Атаки или ошибки, приводящие к неконтролируемому потреблению вычислительных ресурсов и недоступности сервиса.
- **VS Energy Threat Scenario:** Скрипт с ошибкой маршрутизации инициирует серию запросов с SCADA-логами на 100K+ токенов. Раздутый KV-Cache мгновенно заполняет 24GB VRAM RTX 3090 (OOM). Процесс Ollama падает, LLM-сервис недоступен для SOC-инженеров в момент активного инцидента ИБ.
- **Mitigation:** Rate limiting на API Ollama, максимальный лимит токенов на запрос, мониторинг VRAM в реальном времени (nvidia-smi + alerting), watchdog для автоперезапуска сервиса.

## Category 5: Enterprise & SAP Integration
*(9 терминов)*

**47. Tool Calling / Function Calling (Вызов инструментов)**
- **Definition:** Механизм, позволяющий LLM вызывать внешние функции и API вместо генерации ответа самостоятельно.
- **Mechanism:** LLM получает описание доступных функций → решает какую вызвать → возвращает структурированный вызов → получает результат → генерирует финальный ответ.
- **VS Energy Application:** Безопасная интеграция с SAP API: ключи авторизации хранятся в Vault, LLM лишь передаёт параметры (номер договора, период), не зная самих ключей. Решает проблему LLM07 (System Prompt Leakage).

**48. API Gateway (Шлюз API)**
- **Definition:** Промежуточный слой между LLM-системой и корпоративными API, обеспечивающий авторизацию, логирование и rate limiting.
- **Mechanism:** Все вызовы LLM → API Gateway → внутренние системы (SAP, CRM, Billing). Никаких прямых соединений.
- **VS Energy Application:** Обязательный компонент для production LLM-интеграций. Централизованное место для аудита всех запросов агентов к корпоративным системам.

**49. Human-in-the-Loop / HITL (Человек в петле управления)**
- **Definition:** Архитектурный паттерн, требующий подтверждения человека перед выполнением критических действий агентом.
- **Mechanism:** Агент предлагает действие → пауза → оператор одобряет/отклоняет → выполнение.
- **VS Energy Application:** Обязателен для любых write-операций через LLM-агентов: изменения в SAP, команды в SCADA, финансовые транзакции. LLM06:2025 mitigation.

**50. Audit Logging (Журналирование аудита)**
- **Definition:** Запись всех действий LLM-системы: входящие промпты, исходящие ответы, вызовы инструментов, идентификаторы пользователей.
- **Mechanism:** Неизменяемый (append-only) журнал с временными метками и идентификаторами сессий.
- **VS Energy Application:** Требование GDPR и стандартов ИБ для критической инфраструктуры. Необходим для расследования инцидентов (кто запрашивал информацию о SCADA-топологии).

**51. Data Classification (Классификация данных)**
- **Definition:** Разметка данных по уровню конфиденциальности перед загрузкой в LLM-системы.
- **Mechanism:** Уровни: Public → Internal → Confidential → Secret. Определяет: какие данные можно загружать в RAG, какие модели использовать, где хранить.
- **VS Energy Application:** Классификация для VS Energy: SCADA-конфигурации = Secret (только локальная обработка), клиентские договора = Confidential (локальный RAG без облака), публичные регламенты ПУЭ = Public (можно использовать cloud-API).

**52. Orchestration (Оркестрация)**
- **Definition:** Управление последовательностью вызовов LLM, инструментов и агентов для выполнения сложных задач.
- **Mechanism:** Фреймворки: LangChain, LlamaIndex, CrewAI. Определяют граф выполнения, обработку ошибок, передачу контекста между шагами.
- **VS Energy Application:** Lab A5. Оркестрация цепочки: мониторинг Zabbix → LLM-анализ → SAP-запрос → Jira-тикет. Каждый шаг = отдельный tool call с логированием.

**53. RAG vs Fine-tuning Decision Framework**
- **Definition:** Критерии выбора между RAG и Fine-tuning для конкретной задачи.
- **Mechanism:** RAG → когда нужны актуальные факты, новые документы, прозрачность источников. Fine-tuning → когда нужен специфический стиль вывода, формат, синтаксис.
- **VS Energy Application:** Правило: "Факты = RAG, Формат = Fine-tuning". Новые регламенты → RAG. Специфический формат отчётов для НКРЕКП → Fine-tuning.

**54. TCO (Total Cost of Ownership — Совокупная стоимость владения)**
- **Definition:** Полные затраты на внедрение и эксплуатацию решения: инфраструктура + лицензии + персонал + обучение + поддержка.
- **Mechanism:** TCO = CAPEX (капитальные затраты) + OPEX (операционные расходы) за период (обычно 3 года).
- **VS Energy Application:** On-premise стек (RTX 3090 + Ollama) vs cloud API. CAPEX: железо уже куплено. OPEX: электричество ~$50/мес. Cloud API при интенсивном использовании: $500-2000/мес. Break-even: ~3-6 месяцев.

**55. GRC (Governance, Risk, Compliance — Управление, Риск, Соответствие)**
- **Definition:** Система управления рисками и соответствием нормативным требованиям.
- **Mechanism:** Governance = политики и процедуры. Risk = идентификация и митигация рисков. Compliance = соответствие регуляторным требованиям (GDPR, NIS2, галузеві стандарти).
- **VS Energy Application:** Все LLM-решения VS Energy должны проходить GRC-оценку: GDPR для клиентских данных, требования регулятора энергетики (НКРЕКП) для SCADA-данных.

## Category 6: On-Premises Infrastructure
*(7 терминов)*

**56. Ollama**
- **Definition:** Open-source runtime для запуска LLM локально. Обеспечивает REST API совместимый с OpenAI, управление моделями, автоматическую загрузку в VRAM.
- **Mechanism:** Абстракция над llama.cpp. Управляет моделями как контейнерами: `ollama pull`, `ollama run`, `ollama serve`.
- **VS Energy Application:** Backbone on-premise стека. Адрес: `http://192.168.0.128:11434`. Версия: v0.17.4. OLLAMA_NO_CLOUD=1 — обязательная настройка для изоляции от облака.

**57. Quantization (Квантизация)**
- **Definition:** Снижение точности числового представления весов модели для уменьшения объёма VRAM.
- **Mechanism:** FP16 (16-bit float) → INT8 (8-bit integer) → INT4 (4-bit integer). Каждый шаг вдвое снижает VRAM, с нарастающей деградацией качества (perplexity).
- **VS Energy Application — Decision Matrix:**

| Уровень | VRAM на 30B | Качество | Применение |
|---------|-------------|----------|------------|
| FP16 | ~60GB | Эталон | Невозможно на RTX 3090 |
| Q8 | ~32GB | -1-2% | Невозможно для 30B, отлично для 14B |
| Q6 | ~24GB | -2-3% | Граничный для 30B, нет запаса под контекст |
| Q4 | ~18GB | -5-8% | Стандарт для 30B в лаборатории |

**58. KV-Cache (Key-Value Cache)**
- **Definition:** Кэш промежуточных тензоров механизма attention, накапливаемый во время генерации.
- **Mechanism:** Растёт линейно с каждым новым токеном в контексте. Занимает оставшуюся после загрузки весов VRAM. Главная причина OOM при длинных диалогах.
- **VS Energy Application:** Расчёт: qwen3:14b (Q8, 9.3GB) на RTX 3090 (24GB) → 14.7GB под KV-Cache → реальный контекст ~40-60K токенов. Всегда рассчитывать реальный лимит, а не доверять заявленным 128K.

**59. VRAM (Video RAM — Видеопамять)**
- **Definition:** Высокоскоростная память на видеокарте, используемая для хранения весов модели и KV-Cache во время inference.
- **Mechanism:** VRAM ≠ RAM системы. Данные должны быть в VRAM для GPU-ускоренного inference. Переполнение → OOM → падение процесса.
- **VS Energy Application:** RTX 3090: 24GB VRAM — физический потолок. Управление: `nvidia-smi` для мониторинга. Предупреждение: никогда не запускать два 30B-model одновременно.

**60. GGUF Format**
- **Definition:** Бинарный формат хранения квантизованных LLM-моделей (преемник GGML). Используется llama.cpp и Ollama.
- **Mechanism:** Содержит: квантизованные веса, метаданные модели, конфигурацию токенизатора в одном файле. Имя файла кодирует уровень квантизации: `model-Q4_K_M.gguf`.
- **VS Energy Application:** Все модели в Ollama хранятся в GGUF. При загрузке с внешних источников — верификация SHA-256 обязательна (LLM03:2025 Supply Chain).

**61. Air-gapped Deployment (Изолированное развёртывание)**
- **Definition:** Эксплуатация LLM-системы в полностью изолированной от интернета среде.
- **Mechanism:** Все модели предзагружены локально. Нет исходящих соединений для API-вызовов. Обновления — через контролируемый канал.
- **VS Energy Application:** Целевой режим для систем работающих с SCADA-данными и Secret-classified информацией. On-premise Ollama + локальные модели = частичная реализация air-gapped принципа. Ollama v0.17.4: OLLAMA_NO_CLOUD=1 обязателен.

**62. GPU Acceleration (GPU-ускорение)**
- **Definition:** Использование параллельной архитектуры GPU для матричных операций нейронных сетей вместо последовательного CPU.
- **Mechanism:** GPU имеет тысячи параллельных ядер (RTX 3090: 10496 CUDA-ядер). Матричное умножение в LLM — идеально параллелизуемая операция.
- **VS Energy Application:** CPU inference: 2-5 t/s. GPU inference на RTX 3090: 40-80 t/s для 14B-моделей. Разница: пригодность системы для реального использования инженерами.

## Architecture Decision Log

### ADL-01: MoE vs Dense для задач VS Energy

| Критерий | MoE (qwen3:30b, qwen3-coder:30b) | Dense (qwen3:14b, deepseek-r1:14b) |
|----------|----------------------------------|-------------------------------------|
| Скорость (t/s) | ↑ Выше (3-4×) | ↓ Ниже |
| Риск потери деталей | ↑ Есть (router collapse) | ↓ Минимален |
| Применение VS Energy | Генерация кода, документов, рутинный анализ | SOC-анализ, GRC-аудит, SCADA-конфигурации |
| Reasoning задачи | Достаточно | deepseek-r1:14b/32b предпочтительнее |

### ADL-02: Quantization для RTX 3090 (24GB VRAM)

- **FP16** — только для embedding-моделей ≤2B. Максимальная точность векторизации.
- **Q8** — золотой стандарт для 8B-14B моделей. Минимальные потери, большой запас под KV-Cache.
- **Q4** — обязательный стандарт для ~30B моделей. Единственный способ запустить локально. Требует валидации на сложных задачах.

### ADL-03: Правило Antigravity / VS Code

- **Antigravity (Google)** — весь код обрабатывается на серверах Google. **ЗАПРЕЩЕНО** для кода, содержащего реальные данные VS Energy (SCADA-конфигурации, SAP-настройки, PII клиентов, топология сети). Разрешено только для sandbox/учебных задач без реальных данных.
- **VS Code + Continue.dev + Ollama** — 100% локально. **Обязательно** для всего кода, связанного с реальными системами VS Energy.
