## Q1. Звонки диспетчеров по авариям/отключениям
Q1. В ONDO Ukraine диспетчеры ежедневно принимают телефонные звонки от клиентов по авариям и плановым отключениям. 
Данных: ~15K звонков/год, записи звонков есть (аудио + краткие заметки оператора). 
Оцени этот use case по полной схеме: 
(a) подходит ли для AI?, 
(b) три оси feasibility, 
(c) куда попадает в Value/Complexity matrix, 
(d) build vs buy рекомендация, 
(e) топ-3 OWASP LLM угрозы.

**(a) Подходит ли для AI?**
Да, но с правильной постановкой (не «заменить диспетчера», а ассистент/анализ):
- Паттерн сложности: звонки разные по формулировкам, но типы повторяются (авария, плановое отключение, статус работ, жалобы). 
	Этогичный паттерн для классификации, приоритизации, подсказок оператору.
- Данные: ~15K звонков/год — уже приличный объём для:
  - классификации по типам/приоритету (нужна разметка части звонков);
  - извлечения сущностей (адрес, лицевой счёт, тип аварии);
  - ассиста оператору в реальном времени (подсказка скрипта/ответов).
- Приемлемость ошибки:
  - Автоматическое принятие решений по управлению сетью → **нет**, только рекомендации;
  - Подсказки оператору, автозаполнение карточки, аналитика очередей → **да**, приемлемо.
- Повторяемость: 15K/год≈60 звонков/рабочий день — высокая повторяемость, хороший кандидат на AI. 
**Вывод:** Use case подходит для AI, если формулировать его как:
- «AI‑ассистент для диспетчера»: авто‑классификация, подсказка ответа, заполнение карточки, аналитика нагрузки, а не автономный голосовой бот, полностью заменяющий оператора.

**(b) Три оси feasibility**
1) **Technical Feasibility**
Плюсы:
- Исторические данные есть (аудио + заметки), можно:
  - сделать авто‑транскрибацию старых записей (ASR — Automatic Speech Recognition), разметить часть данных и обучить классификатор/модель подсказок;
  - строить модели приоритизации (авария/не авария, «критично/некритично»).
- RTX 3090 + on‑prem позволяют:
  - крутить модели ASR/LLM локально;
  - делать inference без выноса телеком‑данных наружу.
Риски:
- Качество заметок операторов: если они очень короткие/неструктурированные, потребуется серьёзная **data prep** и ручная разметка.
- Для полноценного voice‑бота уровня «говорит сам с клиентом» нужны более сложные стеки (low‑latency ASR+TTS+LLM‑оркестрация), это повышает сложность и требование к latency.
**Оценка:** технически реализуемо как ассистент/аналитика, для полного voice‑бота — High complexity.
2) **Business Feasibility**
- Метрики:
  - Время обработки звонка (AHT — Average Handle Time);
  - Время ответа (ASA — Average Speed of Answer);
  - First Contact Resolution (решение за один контакт).
- Деньги:
  - сокращение времени на оформление/поиск информации → экономия FTE;
  - лучшее качество логирования аварий → меньше ошибок/пропусков, потенциально снижение штрафов/недопоставок.
- Спонсор:
  - типичный спонсор — директор по операционной деятельности/техдиректор + руководитель колл‑центра/диспетчерской.
**Оценка:** бизнес‑потенциал **Medium–High** (много операций, но прямой денежный эффект зависит от тарифной модели, штрафов и SLA).
3) **Organizational Feasibility**
- Нужно:
  - владельца процесса (руководитель диспетчерской/тех-директор);
  - договорённость, что операторы будут пользоваться ассистентом, а не саботировать;
  - кто будет ML‑owner: отдел IT/цифровой трансформации, кто отвечает за:
    - обновление словарей, списков работ, плановых отключений;
    - переобучение модели, если меняется тарифная/техническая политика.
- Риск: если диспетчерам не объяснить ценность и не обучить, система «повиснет» как ещё один экран.
**Оценка:** организационно **Medium**: это не один отдел ИТ, это операционная служба, но не вся компания.

**(c) Value/Complexity matrix**
- **Value:** Medium–High (сокращение FTE, повышение качества, лучше SLA, но не прямой «золотой рудник»).
- **Complexity:** Medium (интеграции с телефонией/CRM, ASR, обучение операторов, процессные изменения).
По матрице:
- Если делаешь **AI‑ассистента для диспетчера** (подсказки, авто‑черновик заметок, аналитика) → Value=High, Complexity=Medium → ближе к **Quick Win / Strategic Light Bet** (P1–P2).
- Если целишься в **полноценного голосового бота, принимающего звонки вместо людей** → Value=High, Complexity=High → **Strategic Bet**, точно не первый проект.

**(d) Build vs Buy рекомендация**
Разделю:
1) **ASR (распознавание речи) + телефония:**
- Это **commodity**, есть масса готовых решений (on‑prem и облачные), в т.ч. с украинским/русским.
- Для критической инфраструктуры важно местоположение данных и соответствие GDPR/NIS2.
- Тут разумен **Hybrid**:
  - использовать готовый ASR (on‑prem или частное облако в ЕС);
  - поверх — свой слой логики и интеграций (классификация, RAG по внутренним регламентам).
2) **Классификация/подсказки/аналитика:**
- Это уже свой **domain‑specific** слой:
  - типы аварий, приоритеты, связи с SCADA/SAP;
  - специфические скрипты общения.
- Здесь логично **Build/Hybrid**:
  - свои модели (на Qwen/DeepSeek), крутящиеся on‑prem;
  - возможно, готовые компоненты для контакт‑центров, но логика классификации и интеграции — своя.
**Итого:**  
- Для общего голосового стека → **Buy/Hybrid** (готовые компоненты + своя оркестрация).  
- Для логики аварий/приоритета/аналитики → **Build/Hybrid**. Полный чистый SaaS‑бот для критической инфраструктуры без контроля над моделью и данными категоричеки не рекомендоан.

**(e) Топ‑3 OWASP LLM угрозы**
Если говорить про ассистента диспетчера / голосового бота:
- **LLM01: Prompt Injection (внедрение инструкций во ввод)**  
  Клиент или атакующий может попытаться «заставить» модель игнорировать системные правила, выдавать скрытую информацию, изменять формат действий.
- **LLM02: Sensitive Information Disclosure (раскрытие чувствительных данных)**  
  Модель может:
  - случайно раскрыть PII (персональные данные клиентов), детали аварий, коммерческую информацию, если они попадают в контекст;
  - в режиме ассистента — «утекать» в текст то, что было в других диалогах (контекстное окно).
- **LLM06: Excessive Agency (избыточные полномочия)**  
  Если агент получает право:
  - автоматически создавать/менять заявки, переключать статусы, инициировать команды в SCADA/CRM/OMS,
  то ошибка/атака может привести к неправильным действиям в операционной системе (например, неверная классификация/маркировка аварии).
(LLM04 Data/Model Poisoning тоже релевантен, если кто‑то сможет подмешать «левые» данные в историю звонков для обучения, но топ‑3 выше — самые острые.)

## Q2. Ты создаёшь TCO для use case «Интеллектуальный поиск по SAP-документации для инженеров (RAG-система)». 
Year 0 CAPEX: data prep = $15K, integration = $10K, dev = $20K. 
Year 1-3 OPEX/year: инфраструктура = $3K, обслуживание = $5K, 0.2 FTE = $12K. 
Посчитай 3-Year TCO. Назови 2 статьи, которые я, возможно, пропустил.

**Дано:**
Year 0 (CAPEX):
- Data prep: 15K
- Integration: 10K
- Dev: 20K
Year 1–3 (OPEX per year):
- Infrastructure: 3K
- Maintenance: 5K
- 0.2 FTE: 12K
**1) Считаем Year 0 CAPEX:**
\(15K + 10K + 20K = 45K\).
**2) OPEX per year:**
\(3K + 5K + 12K = 20K\) в год.
За 3 года: \(20K × 3 = 60K\).
**3) 3‑Year TCO:**
\(TCO_{3y} = CAPEX_{Y0} + OPEX_{Y1-3} = 45K + 60K = 105K.\)
**Ответ:** 3‑Year TCO ≈ **105K USD**.

**2 статьи, которые могли быть пропущены:**
- **Shadow mode operation / Change management**  
  1–2 месяца, когда инженеры работают «по‑старому» и параллельно тестируют новый RAG, плюс время на обучение, документацию, поддержку пилота. 
  Это реальные человеко‑часы, часто несколько FTE‑месяцев.
- **Security & compliance / audit overhead**  
  Для SAP/критической инфраструктуры:
  - аудит архитектуры RAG на предмет LLM02/LLM08;
  - настройка логирования, ролевой модели, ревью со стороны безопасности, возможные внешние аудиты.  
  Это отдельные часы InfoSec/Compliance/аудиторов, которые обычно не считают в TCO, но они есть.

(Дополнительно часто забывают лицензии на сопутствующие компоненты — observability, векторную БД, секрет‑менеджмент, но в запросе только две.)

## Q3. Почему включение threat analysis в каталог use cases (Lab A2), а не в фазу implementation — это не бюрократия, а реальная ценность? 
	Дай конкретный пример для ONDO, где ранний security-анализ изменил бы приоритизацию use case.

Коротко: ранний security‑анализ меняет **приоритеты** и **формат решения**, а значит — портфель проектов и его ROI. Это не бюрократия, а фильтр «не влезть в минное поле без смысла».
### Почему это ценность, а не «бумага»
Проверенный факт: для LLM‑проектов типичные риски — утечки данных, prompt injection, избыточные полномочия — проявляются уже на уровне **архитектуры** и **способа интеграции**, а не только реализации. 
Если security подключается, когда архитектура и выбор вендора уже сделаны, часто приходится либо:
- резать функциональность;
- либо серьёзно удорожать решение (добавлять прокси, фильтры, re‑архитектуру).
То есть поздний security = рост TCO и задержка Time To Market.
### Конкретный пример для ONDO
Представим два use case, оба попали в каталог:
1) **UC1: Внутренний RAG‑ассистент по общим регламентам для ИТ‑поддержки**  
   - Доступ только у внутренних ИТ‑сотрудников;
   - Документы уровня «инструкции, процедуры без критичных SCADA/SAP‑деталей»;
   - Threat analysis:
     - LLM01 Prompt Injection — Medium;
     - LLM02 Sensitive Information Disclosure — Medium;
     - LLM08 Vector & Embedding Weaknesses — Medium.
2) **UC2: RAG‑ассистент для работы с эксплуатационной документацией SCADA/сетевой инфраструктуры (топология, настройки реле, схемы подстанций)**  
   - Доступ у ограниченного круга, но:
   - Документы содержат детали критической инфраструктуры, которые подлежат защите по NIS2/украинским требованиям;
   - Threat analysis:
     - LLM02 Sensitive Information Disclosure — **High/Critical** (утечка схем/настроек);
     - LLM04 Data/Model Poisoning — High (если злоумышленник подменит часть документов);
     - LLM08 Vector & Embedding Weaknesses — High (эксфильтрация через RAG).
Если сделать threat analysis **на стадии каталога (Lab A2)**:
- UC1 остаётся **Quick Win (P1)**:
  - Можно даже позволить себе более лёгкий стек, возможно частично облачный (если без PII/критичных данных).
- UC2 превращается в:
  - либо **Strategic Bet**, требующий:
    - 100% on‑prem, строгой сегментации, отдельного security‑дизайна;
    - серьёзного участия CISO/OT‑безопасности;
  - либо вообще откладывается, если ресурса на безопасную реализацию нет.
В результате:
- Ты не запускатся рисковый UC2 как «ещё один RAG, чего там»;
- Не тратится 6–12 месяцев и деньги на пилот, который потом зарежет служба безопасности;
- Вместо этого фокусируешься на UC1 и других менее рисковых P1, которые реально дадут ROI.
Если бы threat analysis делали **только на этапе implementation**:
- UC2 мог бы пройти в приоритетный список как «очень полезный для эксплуатации»;
- Бюджет заложили бы, пилот запустили;
- На этапе security‑ревью выяснилось бы, что для соответствия NIS2 нужно:
  - менять архитектуру, вендора, добавлять слой анонимизации/маскировки;
  - либо урезать функциональность так, что ценность падает.  
  В итоге — либо удорожание в 1.5–2 раза, либо закрытие проекта → чистый минус к портфелю.
Поэтому включение STRIDE+OWASP‑оценки уже в Lab A2 — это способ **сдвинуть «security как тормоз» → «security как инструмент приоритизации портфеля»**, а не бюрократию.
